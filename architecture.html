Harness Architecture Document

Ops Runbook Harness

An AI agent harness that diagnoses and remediates infrastructure incidents using pre-approved runbooks — with safety constraints the agent cannot override.
Author: Andrew Leat Date: June 12, 2026 Stack: Python · FastAPI · Claude / OpenAI · SQLite · Pydantic

1. Overview

When production systems break, on-call engineers get paged for problems with known fixes. The hard part isn't running the fix — it's knowing whether automation can safely act, verifying the fix actually worked, and catching downstream damage before it cascades.

This harness governs an AI agent that follows pre-approved runbooks to diagnose and remediate incidents. The agent proposes actions; the harness decides what's allowed. Four architecturally separate pillars — each a distinct module — enforce safety, evaluate quality, structure data, and signal failures.

End-to-end architecture. Guardrails wrap the loop. Checkpoints gate each stage. Alarms halt on critical. All four pillars are separate modules.

Design Principles

Pillar Separation

Each pillar is a distinct Python module. The agent never touches guardrail config, checkpoint persistence, or alarm emission directly.

Declared, Not Implicit

Guardrails live in guardrails.yaml, not prompt engineering. Change what's allowed without changing code.

Block First, Ask Second

Any action not on the allow-list is blocked before execution. The AI cannot make things worse — it can only do what's pre-approved.

Swappable Agent

BaseAgent abstract interface. Claude or OpenAI — zero harness changes. The harness is the product, not the model.

2. The Four Pillars

PillarModuleWhat It DoesKey Contract
Guardrails harness/guardrails.py Loads YAML config. Checks every tool call against the allow-list. Enforces turn limits, token budgets, timeouts. Gates production access behind human approval. Returns Allowed, Blocked, or NeedsApproval for every action.
Checkpoints harness/checkpoints.py Evaluates explicit pass/fail criteria at each stage (CP1–CP4). Persists state to SQLite. Enables replay from any checkpoint forward. Each checkpoint returns passed: bool with state snapshot. Replay loads prior state, skips completed stages.
Material Handling harness/material.py Validates incoming alert JSON (Pydantic schema). Structures the outgoing remediation report: actions taken, outcomes, metrics before/after, downstream effects. Invalid input rejected before the loop starts. Output is a complete incident record.
Alarms harness/alarms.py Emits structured signals when something goes wrong. Five named types with severity. CRITICAL alarms halt the loop and trigger human-in-the-loop escalation. Each alarm: {type, context, severity, recommended_action}. CRITICAL = halt. WARNING = log.

3. Key Decisions

Why the agent can't bypass guardrails: The tool registry checks the allow-list before executing any tool. The agent receives a "blocked" result and must choose a different action. This is the core safety guarantee — an unconstrained AI responding to a production incident can take the system down.

Why SQLite for checkpoints: Single-file, zero infrastructure, persists on Railway's volume. Replay is a query, not a re-run. At this scale, it's the simplest correct choice.

Why YAML for guardrails config: Human-readable, version-controllable, diffable. An ops team can review and approve guardrail changes in a PR. The agent never reads this file — only the harness does.

The core safety contract

The agent proposes. The harness disposes. No action executes without passing the guardrail allow-list check. No checkpoint passes without meeting explicit criteria. No critical failure goes unescalated. The model is the engine — the harness is the car.
